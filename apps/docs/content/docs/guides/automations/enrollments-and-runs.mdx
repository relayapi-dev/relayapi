## Enrollment

When an automation's trigger fires for a contact, the matcher creates an `automation_enrollments` row:

```
id                  aten_...
automation_id
automation_version  (immutable — snapshot the enrollment runs against)
contact_id          (null for triggers that don't resolve to a contact)
current_node_id     (moves as the enrollment advances)
state               jsonb — captured inputs + HTTP results + internal markers
status              active | waiting | completed | exited | failed
next_run_at         when the scheduler should next advance (for waits)
enrolled_at / completed_at / exited_at
exit_reason         populated on exit / fail
```

## Status Lifecycle

```
active  → runner picks it up → runs node-by-node
  ├── wait_for_input or smart_delay → waiting
  │       └── resume → active
  ├── reaches end / no outgoing edge → completed
  ├── exit_on_reply fires → exited
  └── handler throws or snapshot missing → failed
```

## Listing Enrollments

```bash
GET /v1/automations/:id/enrollments?status=waiting&limit=50
```

Supports cursor pagination via `cursor` (pass the last row's `id` from the previous page).

```json
{
  "data": [
    {
      "id": "aten_abc...",
      "automation_id": "aut_...",
      "contact_id": "ct_...",
      "current_node_id": "atnd_...",
      "state": { "email": "...", "_pending_input_field": "email" },
      "status": "waiting",
      "next_run_at": null,
      "enrolled_at": "2026-04-17T12:34:56Z"
    }
  ],
  "next_cursor": "aten_abc...",
  "has_more": true
}
```

## Per-Node Run Log

Every node execution writes a row to `automation_run_logs`, including skipped and failed attempts. Call:

```bash
GET /v1/automations/:id/enrollments/:enrollmentId/runs
```

Response:

```json
{
  "data": [
    {
      "id": "atrl_...",
      "enrollment_id": "aten_...",
      "node_id": "atnd_welcome",
      "node_type": "message_text",
      "executed_at": "2026-04-17T12:35:00Z",
      "outcome": "ok",
      "branch_label": "next",
      "duration_ms": 182,
      "error": null,
      "payload": { "message_id": "m_abc..." }
    },
    {
      "id": "atrl_...",
      "node_id": "atnd_ask_email",
      "node_type": "user_input_email",
      "executed_at": "2026-04-17T12:35:00Z",
      "outcome": "wait_for_input",
      "duration_ms": 5
    }
  ]
}
```

`outcome` values: `ok`, `wait`, `wait_for_input`, `goto`, `complete`, `exit`, `failed`.

The endpoint enforces org ownership + verifies the enrollment belongs to the automation in the URL path. It returns 404 if the enrollment is not accessible.

## Debugging

Typical failure paths + what to look for:

- **Handler error** — `outcome: "failed"` + `error` field. Usually "social account not found or has no token" for send nodes, or `custom field '<slug>' not defined` for field writes.
- **Stuck waiting** — `status: waiting` + `next_run_at` in the past. Check the `automation_scheduled_ticks` table for pending ticks; the cron sweep claims only `BATCH_SIZE` per tick.
- **Never enrolled** — trigger matcher did not find a match. Check the automation is `active`, `published_version` is non-null, trigger type and keyword config match the payload, and filters pass.
- **Re-entry blocked** — contact already has an enrollment; either set `allow_reentry: true` or wait for `reentry_cooldown_min`.

## Snapshot Isolation

Every enrollment records `automation_version` at creation time and reads from `automation_versions.snapshot` for that version. Editing the automation while enrollments are in flight does not break them — they continue running against the published snapshot they started on.

Re-publishing bumps the version; new enrollments start on the new version, old ones finish on the old one.
